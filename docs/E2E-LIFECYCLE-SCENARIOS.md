# E2E·수동 테스트 시나리오 — 프로젝트 생성부터 관리 종료까지

> **버전**: v1.2  
> **작성일**: 2026-04-14  
> **목적**: 웹(`apps/web`)·API·모바일(`apps/mobile`)을 아우르는 **엔드투엔드 검증 흐름** 정의. 자동화(E2E) 스크립트 초안·UAT 체크리스트로 활용한다.  
> **연계**: [WIREFRAME-SPEC.md](WIREFRAME-SPEC.md) · [PRD.md](PRD.md) · [TASKS.md](TASKS.md) T-700

## 1. 범위·전제

| 항목 | 내용 |
|------|------|
| **대상** | 로그인 이후 **프로젝트 생성 → 운영 → 스프린트·이슈 관리 → 릴리즈·감사 → 운영 종료(세션/데이터 정리)** 까지 |
| **환경** | API·웹이 동일 베이스 URL에서 동작. 로컬 예: 백엔드 `http://localhost:8080`, 웹 Vite 프록시 또는 `API_BASE_URL` 정의 |
| **역할** | 최소 **ADMIN(프로젝트 창립자)** 1명, **MEMBER** 1명(멤버 초대·권한 검증 시) |
| **데이터** | 시나리오마다 **프로젝트 키·이슈 키**를 메모해 두고, 실패 시 재현에 사용 |

**로컬 시드 계정(개발 프로파일)**: H2 시드에 `admin@local.test` 등이 있을 수 있다. 운영형 검증은 별도 계정을 생성해 사용한다.

## 2. 공통 절차·식별자

- **웹 경로**: 앱 라우트는 `App.tsx` 기준 — `/project/:projectKey/...`, 이슈 상세는 `/issue/:issueKey`.
- **성공 기준**: HTTP 2xx, UI에 오류 토스트 없음, 새로고침 후에도 데이터가 일관되게 보임.
- **실패 시**: 브라우저 네트워크 탭·API 응답 본문(`message`, `code`) 캡처.

## 3. 시나리오 (단계별)

### S-01 — 최초 로그인·세션

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-01a | 정상 로그인 | `/login`에서 유효한 이메일·비밀번호 제출 | 홈(`/`) 또는 이전 `from` 경로로 이동, GNB에 사용자 맥락 표시 |
| S-01b | 무효 자격증명 | 잘못된 비밀번호 제출 | 로그인 실패 메시지, 토큰 미발급 |
| S-01c | 보호 라우트 | 로그아웃 상태에서 `/project/DEMO/backlog` 직접 접근 | `/login`으로 리다이렉트 |

**모바일**: 동일 계정으로 로그인 후 토큰 유지·API 호출 성공.

---

### S-02 — 프로젝트 생성(관리 시작)

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-02a | 프로젝트 생성 | 홈(`/`) **새 프로젝트** 양식에서 키(대문자 2~10자)·이름·설명·보드(SCRUM/KANBAN) 제출 | 목록에 노출, 해당 키로 `/project/{key}` 이동·접근 가능 |
| S-02b | 키 중복 | 동일 키로 재생성 시도 | 4xx 및 사용자에게 이해 가능한 오류 |
| S-02c | 프로젝트 홈 | `/project/{key}` 진입 | 프로젝트 컨텍스트 로드, 사이드바에 백로그·보드 등 메뉴 활성 |

---

### S-03 — 팀·권한(선택·2인 검증)

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-03a | 멤버 추가 | `설정`(`/project/{key}/settings`)에서 다른 사용자 초대/추가(정책에 맞는 API·UI) | 멤버 목록에 표시 |
| S-03b | 역할 차이 | MEMBER로 로그인해 ADMIN 전용 메뉴(예: Audit) 접근 | 숨김 또는 403에 준하는 동작 |

---

### S-04 — 이슈·백로그 (수요 정리)

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-04a | 이슈 생성 | `/project/{key}/issues/new`에서 STORY·TASK 등 타입, 요약·필수 필드 입력 후 저장 | 이슈 키 발급, 상세 화면으로 이동 |
| S-04b | 백로그 노출 | `/project/{key}/backlog` | 생성한 이슈가 리스트에 보임 |
| S-04c | 순서 변경 | DnD로 우선순위 변경 후 저장/자동 반영 확인 | 새로고침 후 순서 유지 |
| S-04d | 스프린트 미배정 상태 | 아직 스프린트에 넣지 않은 이슈 | 백로그에만 존재, 보드에는 정책에 따라 미표시 또는 Backlog 컬럼 |

---

### S-05 — 스프린트 계획·기동

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-05a | 스프린트 생성 | `/project/{key}/sprints`에서 스프린트 생성(이름·목표·기간 등) | 목록에 `PLANNED` 또는 정책상 초기 상태 |
| S-05b | 스프린트 시작 | 시작 액션 | 활성 스프린트 1개 제한 등 비즈니스 규칙 준수 |
| S-05c | 백로그 → 스프린트 | 백로그에서 이슈를 스프린트에 배정 | 스프린트 상세·보드에서 조회 가능 |

---

### S-06 — 실행 — 스크럼·칸반·워크플로

| ID | 목적 | 절차 | 기대 결과 |
|----|------|------|-----------|
| S-06a | 스크럼 보드 | `/project/{key}/board`에서 컬럼(상태) 확인 | 스프린트 이슈가 컬럼에 표시 |
| S-06b | 드래그 전환 | 허용 전이로 카드 이동 | 상태 변경, 이슈 상세와 일치 |
| S-06c | 금지 전이 | 워크플로 위반 전이 시도 | 409류·토스트(`WORKFLOW_VIOLATION` 등) |
| S-06d | 칸반·WIP | `/project/{key}/kanban`에서 WIP 한도 근처로 카드 배치 | 한도 초과 시 정책대로 차단·경고 |
| S-06e | 이슈 상세 | `/issue/{issueKey}`에서 메타·설명·전환·첨부 | 저장·표시 정상 |

**모바일**: 이슈 목록 → 상세 → 스프린트 보드에서 터치 전환(아카이브 이슈는 전환 제한 등 정책 확인).
